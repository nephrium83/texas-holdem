# Code Audit — 2026-07-17

Audit of `main` @ `e4e6413`, with fixes applied in the following commit.
Supersedes the triage in `docs/AUDIT.md` (2026-07-16); the three
CRITICALs from that pass were confirmed fixed, and this pass covers the

Method: full static read, `pyflakes`, and the whole test suite
(now 76 tests, all green — 27 engine, 38 shuffle/p2p, 11 new security).

## Summary

Single-player is solid. The multiplayer stack had one live crash and one
real hole in its security model; both are now fixed, plus two supporting
issues. Two things the first-pass audit flagged turned out to be already
implemented (noted below) — the static read understated them.

## Fixed in this pass

### C-2 · Join-path crash (`NameError`) — FIXED
`sess` was created inside `_do_connect` but referenced from the sibling
nested function `_handle_start`, which never received it. Any joining
client raised `NameError` the moment the host started the game, and the
M-7 man-in-the-middle check that lived in that block never ran.
Fix: share the session via a `_sess_ref` cell (matching the file's
existing `_conn_id_ref` / `_peer_id_prefix_ref` pattern); the M-7 check
is now fail-closed (aborts if the session or host key is missing).
*(Committed in `5de7ba3`.)*

### C-1 · Signatures were opt-out — FIXED
The receiver verified signatures only `if "sig" in msg`, so an attacker
could omit the field and be accepted unverified — and most control and
game-state messages (built directly in `session.py`) were never signed
at all, because only 4 of ~20 send sites went through `wire.pack`.
Fix: signing is now centralized in `transport._sign_frame`, applied to
every peer-bound message in `_send_to`, so **all** send sites are covered
without touching each call site. `_read_msg` now **requires** a valid
signed envelope (via `wire.unpack`) for every peer message; the sole
exception is relay-control frames (`relay_join`), which are addressed to
the relay server, never dispatched to a session, and carry no game
authority. Seat/pubkey binding (below) was already present.
Verified: unsigned rejected, signed accepted, tampered payload rejected,
relay frames exempt, no double-wrapping.

### hash-chain regression from C-1 — FIXED
Enforcing signatures activated the previously-dormant M-11 chain check
(unsigned messages had carried no `hash`/`prev`). Because senders still
emit `prev="0"*64`, strict enforcement would have dropped every peer's
2nd-onward message. Fix: record each message hash for future
verification, but only *enforce* linkage when a real (non-genesis) `prev`
is present — so the chain becomes correct once per-peer sequencing is
threaded (Phase 1) without breaking the game today.

### H-1 · Shuffle "public verifiability" was broken — FIXED
`record_reveal` discarded the nonce, and both broadcast paths substituted
the *commit* into the `nonce_hex` field, so any peer recomputing
`SHA256(seed‖nonce)` failed — the advertised "anyone can verify the
deck" property did not hold, and the peer-side verifier was an empty
`pass`. Fix: retain the nonce (`_nonces`), emit the real nonce in
`reveals_snapshot()`, and implement `_on_shuffle_reveal_collect` to
actually verify every reveal against its commit, derive the deck
independently, and fire a new `on_shuffle_cheat` callback on mismatch.
Verified: snapshot nonces verify against commits, host and peer derive
identical decks, a tampered seed triggers the cheat path.

## Already implemented (audit overstated these)

- **Seat/pubkey binding (was flagged under C-1):** `handle_game_action`
  already rejects a message whose `payload.seat` doesn't match the
  `conn_id` it arrived on (`_seat_order[seat] != conn_id`, line ~723),
  alongside a valid-action check and a turn check. With C-1 now verifying
  signatures, the action path is sound.
- **Hash chain (H-3):** the receive-side linkage check (M-11) already
  existed; it just wasn't threaded on the send side. Reframed above.

## Remaining (tracked, not blocking)

- **H-2 · Host learns every hole card.** `_host_finalise_shuffle`
  computes the plaintext deck, so a malicious host sees all hole cards;
  per-recipient X25519+AES-GCM only hides them from *other peers*. This
  is a trusted-dealer-with-encrypted-delivery model, **not** the mental
  poker in `docs/MULTIPLAYER.md`. Acceptable as an interim step, but the
  shuffle docstring's privacy claim should be qualified, and the real fix
  is the Phase-2 threshold shuffle (no plaintext deck on any node).
- **H-3 (send-side chain), H-4 (`holdem.p2p` ships in the wheel):**
  verify the packaging `packages` list includes the subpackage.
- **LOW nits swept:** removed dead imports/locals in `session.py`
  (`field`, `my_commit_hex`, `identity as _id`, `encrypt_hole_cards`) and
  `shuffle.py` (`Optional`, `Tuple`). `onboarding.py`'s `_session`
  re-export left in place (intentional singleton bridge).
